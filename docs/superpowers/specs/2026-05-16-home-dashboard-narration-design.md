# Spec: Real-Time Home Dashboard + Hierarchical Narration

**Date:** 2026-05-16
**Status:** Draft ‚Äî pending review

## Objective

Replace the `/` home page with a **single-screen, real-time dashboard** that lets
someone watching understand, at a glance, what the trading bot is doing and how
it is performing ‚Äî without scrolling and without trading knowledge.

Three things are primary:

1. **P&L** ‚Äî how much money the bot has made or lost.
2. **Live activity** ‚Äî what the bot is doing right now, updating in real time.
3. **Narration** ‚Äî a plain-language recap of what has happened, at a
   granularity that adapts to the time range and can be **zoomed in**: months ‚Üí
   weeks ‚Üí days ‚Üí 6-hour blocks.

Signals are **secondary** ‚Äî collapsed into a one-line summary that expands on
demand.

Success looks like: a non-technical observer opens `/`, sees the bot's P&L and
current action immediately, reads a paragraph explaining the last period, and
can drill from "this month" down to "this 6 hours" to see finer detail.

## Tech Stack

Existing stack ‚Äî no new frameworks:
- `api`: NestJS 10, Prisma 5, `@nestjs/schedule`-style cron (the existing
  `Scheduler`), RxJS for SSE, Claude/OpenAI adapters.
- `web`: Next.js 14, React Query 5, the typed `lib/api-client.ts`.

## Design

### Component 1 ‚Äî Real-time delivery: SSE `/events`

A new `EventsModule` in `api` exposing `GET /events` as an SSE stream
(`@Sse`, same pattern as `/backtest/stream`).

- An `EventsService` holds an RxJS `Subject<AppEvent>`.
- `TradingService` and the pipeline emit into it: `cycle_started`,
  `decision_made`, `trade_opened`, `trade_closed`, `signals_ingested`.
- The `EventsController` `@Sse('/events')` returns the Subject as an Observable.
- `web` opens one `EventSource` (a `useAppEvents` hook). On each event it
  updates the React Query cache via `queryClient.setQueryData` /
  `invalidateQueries`, so existing `useQuery` consumers refresh instantly.

Rationale for SSE over polling: the bot acts on a cron ‚Äî events are sparse and
bursty. Pushing the moment something happens beats polling every few seconds
and is cheaper when idle.

### Component 2 ‚Äî Hierarchical narration

New `Narration` Prisma model:

```
Narration {
  id          String   @id @default(cuid())
  granularity String   // '6h' | 'day' | 'week' | 'month'
  periodStart DateTime
  periodEnd   DateTime
  summary     String   // LLM-written plain-language recap
  assessment  String?  // brief judgment: did the bot behave sensibly?
  stats       Json     // { pnl, trades, wins, losses, winRate }
  createdAt   DateTime @default(now())
  @@unique([granularity, periodStart])
  @@index([granularity, periodStart])
}
```

A `NarrationModule` with a `NarrationService` and a cron-driven generator:

- **Finest level ‚Äî `6h`:** a cron job (every 6 hours) reads the trades and
  stored decisions in that window and asks the LLM to write a `summary` plus a
  short `assessment` ("was this sensible?") and compute `stats`.
- **Roll-ups ‚Äî `day` / `week` / `month`:** generated by summarizing the
  *child narrations*, not raw trades. A `day` summarizes its four `6h`
  narrations; a `week` summarizes its `day` narrations; a `month` its `week`s.
  Cheaper, and consistent with the zoom hierarchy. Roll-ups run on period
  boundaries (a `day` job at UTC midnight, etc.).
- Missing periods are backfilled lazily: a roll-up with no children for a past
  period generates directly from raw trades as a fallback.

HTTP: `GET /narrations?granularity=<g>&from=<iso>&to=<iso>` ‚Üí `Narration[]`.

"Validates what the bot has done" is the `assessment` field ‚Äî the narration
both describes and lightly judges (e.g. "held through a 4% dip rather than
panic-selling ‚Äî consistent with the stop-loss settings").

## Code Style

Match the existing codebase ‚Äî NestJS providers, RxJS Observables for SSE,
React Query hooks in `lib/queries.ts`, plain-TS shared types in
`api/src/common`. Example (narration HTTP shape, mirrored in `api-client.ts`):

```ts
export interface Narration {
  id: string
  granularity: '6h' | 'day' | 'week' | 'month'
  periodStart: string
  periodEnd: string
  summary: string
  assessment: string | null
  stats: { pnl: number; trades: number; winRate: number }

```

## Testing Strategy

- `vitest`, tests beside code in `packages/api/tests`.
- Unit: narration roll-up logic (period boundary math, child‚Üíparent grouping),
  stats computation, granularity selection from a data span.
- Unit: `EventsService` fan-out (subscribe, emit, receive).
- The LLM call itself is mocked (as the existing adapter tests do).
- Manual: open `/`, run a paper cycle, confirm the activity feed updates live
  and the narration zoom works.

## Boundaries

- **Always:** run `pnpm test` before commit; keep `/` rendering when the API or
  narration data is absent (loading/empty states); cap LLM narration cost.
- **Ask first:** the Prisma schema change (new `Narration` model + migration);
  narration cadence if it would exceed ~6 LLM calls/day.
- **Never:** block the trading loop on narration generation; commit secrets;
  remove the paper-mode guard.

## Success Criteria

- [ ] `/` fits one viewport at 1440√ó900 with no scroll.
- [ ] Opening `/` during a paper cycle shows the bot's action in the live feed
      within ~1s of it happening (SSE, not a poll tick).
- [ ] The narration panel shows a written recap + assessment and can zoom
      month ‚Üí week ‚Üí day ‚Üí 6h, each level showing its own narration.
- [ ] P&L (total + today) is the most prominent element.
- [ ] Signals are one collapsed line by default, expandable.
- [ ] Narration generation never delays a trading cycle.
- [ ] `pnpm build` and `pnpm test` pass.

## Open Questions

1. **Finest granularity ‚Äî `6h` or `day`?** `6h` gives a fresher "what just
    happened" recap but is ~4 LLM calls/day before roll-ups. Spec currently
    assumes `6h`.
2. **Backfill depth** ‚Äî when the feature ships, do we generate narrations for
    *historical* trades already in the DB, or only from launch forward?
3. **Live activity feed history** ‚Äî does it show only events since the page
    opened, or load the last N decisions on mount then stream?
